# 核心行为准则与修改边界

你当前在一个已成型的项目中协助我。为了保证项目稳定性，请严格遵守以下权限和修改原则：

1. **以现有代码为绝对基准**
   - 当前项目的文件结构、框架设计以及我已经写好的代码是绝对标准。
   - **严禁**进行任何形式的全局重构、框架设计更改或大范围重写。
   - 尤其是我已经配置好的核心路由、中间件（Middleware）、依赖系统以及数据库连接等底层架构，除非我明确要求，否则绝对不允许触碰。

2. **最小侵入性修改（局部修改）**
   - 所有的代码调整必须是“增量式”或“精准局部”的。
   - 仅限在你职责范围内，修改我明确指出需要排查 Bug 或新增逻辑的特定函数、类或代码块。不要随意优化你认为“不够优雅”的其他无关代码。

3. **严格禁止的越权行为**
   - **禁止**擅自创建、移动、重命名或删除现有文件和目录。
   - **禁止**擅自修改任何依赖配置文件（如 requirements.txt、package.json 等）或引入新的第三方库。

4. **强制确认机制（Do Not Act Without Permission）**
   - 如果你评估后认为，解决当前问题【必须】跨文件修改，或者【必须】改动我原有的核心逻辑，你**必须停止修改**。
   - 你需要先向我输出你的分析结果和预期的修改方案，**只有在我明确回复“同意”后，你才能生成并应用代码**。

5. **代码风格的一致性**
   - 新增的代码必须完全贴合我现有的命名规范、注释习惯和错误处理逻辑。不要强行套用你默认的“最佳实践”来覆盖我的代码习惯。

请回复“已了解项目边界”，随后等待我的具体指令。
